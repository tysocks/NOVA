Three Approaches

All three include the Ultra Minimal UI redesign. They differ in how deep the performance fixes go.

A

Client-Only Optimization

No backend changes. Fix the hot paths entirely in the browser.

  • Real-time fix (0.5fps → ~30fps): Pre-compute traces once when real-time starts, store x/y as Float64Arrays, use Plotly.restyle() per-frame to update only the visible slice — no full rebuild each tick
  • Hot loop fix: Pre-cache __ts (ms timestamp) on every row at load time — eliminates new Date() calls inside applyFreq, plotRows, sorting, and masking
  • Debounce saveState: Currently serializes everything on every click; debounce to 150ms
  • Lightweight list rendering: DOM virtualization for channel/test lists >100 items
  • Ultra Minimal UI redesign

Pros

  • No backend risk
  • Real-time becomes genuinely usable
  • Fast to ship

Cons

  • Still transfers raw 100k–1M rows over the wire on every Chooch
  • DB query + JSON serialization stays slow for large tests
B

Client + Server Downsampling (recommended)

Adds LTTB downsampling to the backend. Cuts data transfer by 100x for typical loads.

  • Everything in Option A
  • LTTB downsampling endpoint: New max_points param on /api/timeseries (default 5000 per channel). Largest-Triangle-Three-Buckets is visually lossless — a 500k-point trace is indistinguishable from a 5k-point one on screen
  • Result: Chooch goes from transferring ~80MB of JSON to ~800KB. DB query unchanged but serialization and network become negligible
  • Full-res export still works: CSV export bypasses downsampling and exports raw rows already in memory (or re-fetches with max_points=unlimited)
  • Ultra Minimal UI redesign

Pros

  • Load time drops dramatically
  • Real-time becomes smooth
  • UI interactions snappy
  • Visually indistinguishable from raw data

Cons

  • Small backend change (Python LTTB, ~30 lines)
  • User loses sub-pixel detail (irrelevant on screen)
C

Option B + Web Worker Processing

Moves mask/calc computation off the main thread. Eliminates UI freezes during heavy processing.

  • Everything in Option B
  • Web Worker: computeCalculatedRows(), applyMasks(), and applyFreq() moved off main thread — UI stays interactive while data crunches
  • Transferable TypedArrays: Data shared with Worker via zero-copy transfer, not JSON clone
  • Biggest win when masks + calculated channels are active on 500k+ rows
  • Ultra Minimal UI redesign

Pros

  • Fully non-blocking UI in all scenarios
  • Best long-term architecture

Cons

  • Largest refactor — Worker/main thread comms adds complexity
  • State management more involved
  • Marginal gain if LTTB keeps row counts low
